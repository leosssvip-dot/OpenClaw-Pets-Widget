# 桌宠动画方案推荐

在多次用「SVG 分组 + 手写 GSAP」做和尚敲木鱼后，可以明确：**手写骨骼/时间轴不是适合桌宠的长期方案**。本文说明原因，并给出更科学、可维护的替代方式。

---

## 一、当前方案（SVG + GSAP）为什么不好

| 问题 | 说明 |
|------|------|
| **层级与轴心强耦合** | 棍子、手臂必须是「父子」或完全同轴旋转，否则就脱手。改 SVG 结构要同步改代码。 |
| **每只角色一套手写逻辑** | monk 要写 `buildMonkWorkingTimeline`、`O_MONK4`、`getMonkRig`；lobster/cat 又要重写一套，难以复用。 |
| **设计师与开发脱节** | 设计师改 SVG 分组/ID，开发要改选择器、旋转中心、角度常量，协作成本高。 |
| **漂移与累积误差** | 长时间运行后 transform 容易漂移，需要 fromTo、clearProps 等补救，治标不治本。 |
| **状态机在代码里** | 三态（idle/working/offline）用 if/switch 切 timeline，状态一多就难维护。 |

结论：**动画逻辑不应写在业务代码里，而应在「动画资产」里完成**，应用只负责「加载资产 + 设状态」。

---

## 二、推荐方式：动画由「制作工具」产出，应用只做「播放 + 状态」

适合桌宠的通用做法是：

1. **在专业工具里**：画角色、绑骨骼/图层、做状态机、做过渡、打事件（如敲击瞬间）。
2. **导出单一资产**：一个文件（如 `.riv` 或一组 Lottie JSON）包含所有状态与时间轴。
3. **应用侧**：加载该资产，根据 `activity` 设一个输入（如 `status`），必要时监听事件（如 `strike`）触发粒子。

这样：

- 骨骼、轴心、层级都在**编辑器**里调，不会出现「棍子脱手」「手没动」这类结构问题。
- 换角色只需换资产，**不写新 timeline 代码**。
- 设计师迭代动画不需要改 TS/GSAP，只需重新导出资产。

## 二（补充）、骨骼动画最合适吗？有哪些方案？

**是的**：对桌宠这种「有肢体、道具、需要自然联动」的角色，**骨骼动画**最合适——在编辑器里绑骨骼、调轴心、做状态与过渡，运行时只负责切状态和播事件，不会出现手写 GSAP 时的脱手、漂移、每角色一套代码等问题。

**骨骼动画不只有 Rive**。主流 2D 骨骼方案对比如下：

| 方案 | 编辑器 / 授权 | Web 运行时 | 状态机 / 事件 | 适合桌宠 | 备注 |
|------|----------------|------------|----------------|----------|------|
| **Rive** | 在线 [editor.rive.app](https://editor.rive.app)；免费创作，**导出需 Cadet $9/月/席** | ✅ 官方 React/Canvas/WebGL 运行时 | ✅ 内置状态机 + 输入 + 时间轴 Event | ✅ 很合适 | 状态机与事件原生支持，与桌宠三态+敲击事件契合；本项目已接 |
| **Spine** | 桌面软件；**买断** 约 $69（基础）/$329（专业） | ✅ 官方 [spine-ts](https://github.com/EsotericSoftware/spine-runtimes)（WebGL/Canvas），spine-player 可直接嵌页 | ✅ 动画状态、事件轨道、可代码控制 | ✅ 很合适 | 功能最全（IK、网格变形、换装）；偏游戏向，生态成熟 |
| **DragonBones（龙骨）** | 桌面/在线；**完全免费开源** | ✅ 官方 [DragonBonesJS](https://github.com/DragonBones/DragonBonesJS)，MIT；另有社区 WebGL 独立渲染器 | ⚠️ 有动画切换与事件，无「状态机」级抽象，需代码里切动画名 | ✅ 合适 | 国内常用、中文界面；官方维护不如 Spine 活跃 |
| **Spriter** | 桌面软件；有免费版 | ⚠️ 无官方 Web 播放器，需用社区 [spriter.js](https://github.com/flyover/spriter.js) 等 | ⚠️ 格式支持动画与事件，需自行封装状态逻辑 | ⚠️ 可用 | 格式开放，生态以游戏引擎集成为主，纯 Web 要自己接 |

**简要结论**：

- **要状态机 + 事件、且接受订阅制**：选 **Rive**（本项目已集成）。
- **要买断、功能最强**：选 **Spine**（需另接 spine-ts 或 spine-player）。
- **要零成本、开源**：选 **DragonBones**（需在代码里按 activity 切动画名并监听事件）。

三种都是「真骨骼」方案，都能解决桌宠的骨骼、轴心、多态与敲击事件需求；按预算和是否已有 Rive 集成来选即可。

## 二（再补充）、用 Rive 时：做一个动画代码控制，还是做三态？

**结论：代码能控制动画——通过改 `status` 决定当前播哪一段；Rive 里需要事先做好多段动画和状态机，用来「响应」这个数字。**

- **代码控制什么**：根据业务的 `activity`（idle / working / offline 等）设置 Rive 的 **`status`** 输入（0、1、2、3、4、5）。  
  一旦你 `statusInput.value = 2`，Rive 会立刻切到 working 对应的那段动画。**所以是代码在决定「现在播哪一态」**，这就是代码对动画的控制。
- **Rive 里要提供什么**：  
  1. **多段动画**：idle、working、offline 等，每段是独立时间轴（可 loop）。  
  2. **一个 State Machine**（`PetStateMachine`），带 Number 输入 `status`。  
  3. **状态机逻辑**：定义好「status==0 播哪段、==2 播哪段、==5 播哪段」以及过渡方式；这样代码一改 `status`，Rive 就自动切动画。

总结：**代码负责「选状态」**（改 `status`），**Rive 负责「每个状态长什么样、怎么切」**。两者配合：代码完全能控制当前播什么，只是动画内容和过渡规则写在 .riv 里，不用写在业务代码里。

详见 [Rive动画接入操作指南.md](./Rive动画接入操作指南.md) 里的「创建 3 个动画」「创建 State Machine」和 status 映射表。

## 三、方案对比（按推荐顺序）

| 方案 | 制作方式 | 产出物 | 应用侧 | 成本 | 适合场景 |
|------|----------|--------|--------|------|----------|
| **Rive** | 在 [editor.rive.app](https://editor.rive.app) 里画矢量、绑骨骼、做状态机、打 Event | 一个 `.riv` 文件 | 加载 .riv，设 `status` 输入，监听 `strike` 事件 | 正式发布需 Cadet $9/月/席（可导出）；运行时免费 | **首选**：多状态、需要事件、希望一个文件搞定 |
| **Lottie** | AE / LottieFiles Creator / SVGator 等做动画 | 每态一个 .json（idle/working/offline） | 按 activity 切对应 JSON 播放，用 `onEnterFrame` 等做 strike | 免费 | **免费替代**：接受多文件，用帧事件或循环完成触发粒子 |
| **SVGA** | AE / Flash 导出 → SVGA 转换器 | 每态一个 .svga（或一个文件多段） | 按 activity 切换播放，`loop`、`onEnd` 等控制，可监听 `process` 做 strike | 免费、开源 | **可选**：与 Lottie 类似，国产、体积小、Web 有现成播放器 |
| **腾讯 VAP** | AE 等导出带透明通道的 MP4 + JSON 配置 | 每态一个 .mp4 + .json（或融合在一个 MP4 内） | 按 activity 切换视频，`loop`、`onVideoComplete`、帧事件 | 免费、开源 | **可选**：偏特效/粒子；做角色 = 预渲染视频，无骨骼、可循环 |
| **序列帧 / 精灵表** | AE / 任何工具导出 PNG/WebP 序列或 sprite sheet | 一组图片 + 元数据（帧数、FPS） | 按索引或时间戳切帧绘制 | 免费 | 像素风、或「只动不绑骨」的简单动画 |
| **SVG + GSAP（当前）** | 手写 timeline、调 svgOrigin、改 SVG 分组 | 无独立资产，逻辑全在代码里 | 直接写死在各角色的 Renderer 里 | 免费但维护贵 | **仅作兜底**：没有 Rive/Lottie 资产时临时用 |

## 三（补充）、SVGA 与腾讯 VAP 是否适合桌宠

### SVGA

- **是什么**：跨平台动画格式，类似 Lottie。设计师用 **After Effects / Animate CC (Flash)** 做动画，用官方/社区**转换器**导出 `.svga`；Web 端用 [SVGAPlayer-Web](https://github.com/svga/SVGAPlayer-Web) 或 [svga-web](https://www.npmjs.com/package/svga-web) 等播放。
- **特点**：开源免费、体积相对小、支持循环（`loop`）、起止帧（`startFrame`/`endFrame`）、事件（如 `end`、`process`），多状态 = 多个 .svga 或一个文件内多段由代码切换。
- **适合桌宠吗**：**可以**。和 Lottie 的用法几乎一致：idle/working/offline 各一个 .svga（或一个 .svga 里三段），根据 `activity` 切源或切段落；在 `end` 或 `process` 里根据进度触发敲击粒子即可。若团队更熟悉 AE + 国内生态，SVGA 是 Lottie 的等价替代选项。

### 腾讯 VAP（Video Animation Player）

- **是什么**：腾讯开源的**带透明通道的视频动画**方案。AE 等导出 **MP4（RGB + Alpha 存成约定格式）**，配合 JSON 配置；播放端用 [video-animation-player](https://www.npmjs.com/package/video-animation-player) 等解码并合成透明视频。
- **特点**：高压缩、可走硬件解码、适合**复杂特效、粒子、融合动画**（如礼物动效、头像嵌入）。支持循环、结束/帧事件。
- **适合桌宠吗**：**技术上可以**，但更偏「预渲染好的动画视频」：三态 = 三个 VAP 文件（或一个 MP4 内多段），没有骨骼/状态机，不能像 Rive 那样在运行时改状态机。适合「整段播完」的循环角色动画；若要做「敲击瞬间打点触发粒子」，需依赖帧事件或时间戳。更适合作为**特效/氛围**（如功德爆发、升级动效），角色本体仍建议 Rive/Lottie/SVGA。

### Adobe Animate 可以吗？

**可以。** [Adobe Animate](https://www.adobe.com/products/animate.html)（原 Flash）本身支持**骨骼工具（Bone tool）**和 **IK 反向运动**，可以做「手—臂—身体」这类层级联动，适合做桌宠角色动画。

- **导出链**：  
  - **→ SVGA**：官方支持的路径。用 SVGA 提供的**转换器/扩展**（如 [SVGA 扩展](https://www.talkae.com/svga-converter-for-after-effects.html) 等），在 Animate 里做好动画后直接导出 `.svga`，Web 用 SVGAPlayer 播放。多态就做多个 .svga 或一个文件内多段，由代码按 `activity` 切换，和文档里 SVGA 用法一致。  
  - **→ Lottie**：Animate **没有**内置 Lottie/Bodymovin 导出；若要坚持 Lottie，一般用 **After Effects + Bodymovin**。  
  - **→ 序列帧 / 视频**：Animate 可导出逐帧图或视频，再走序列帧或 VAP 等方案。

**结论**：用 **Animate 做动画 + SVGA 导出** 是可行且顺畅的一条路，适合已经用 Animate 或 Flash 工作流的设计师；桌宠三态就做三个 SVGA 或一个多段，再接 `SvgaPetRenderer` 和事件即可。

### 小结

| 方案 | 适合做桌宠角色本体？ | 说明 |
|------|----------------------|------|
| **SVGA** | ✅ 适合 | 与 Lottie 同思路：多文件/多段 + 代码切状态，免费、有 Web 播放器，可接 strike 事件。**Adobe Animate 可直接导出 SVGA。** |
| **VAP** | ⚠️ 可用但非首选 | 能做循环角色动画；更适合复杂特效/粒子，角色建议优先 Rive/Lottie/SVGA。 |

## 四、推荐路线（科学且可落地）

1. **短期（当前版本）**  
   - 保留现有 **SVG + GSAP** 作为**兜底**：没有 .riv / Lottie 时仍能显示桌宠。  
   - 不再为 monk / 其他角色**新增**复杂手写 timeline；若有小修，只做最小改动（如 fromTo 防漂移）。

2. **中期（推荐主路径）**  
   - **以 Rive 为桌宠主方案**：  
     - 在 Rive 里重做 monk（可导入现有 SVG 再绑骨骼），做 idle / working / offline 三态 + 状态机 + `strike` 事件。  
     - 导出 `monk.riv`，在 `pet-engine` 里配好 `RIVE_ASSETS['monk'] = '/assets/pets/monk.riv'`。  
     - 应用侧已有 `RivePetRenderer`，只需有 .riv 即自动走 Rive，不再走 `SvgBonePetRenderer`。  
   - 若暂时不想为 Rive 付费（Cadet 才能导出）：  
     - 用 **Lottie**：在 LottieFiles Creator / SVGator 等做三态动画，导出 3 个 JSON，配置到 `LOTTIE_ASSETS`，用现有 `LottiePetRenderer`。  
     - 或用 **SVGA**：AE/Flash 做动画 → 转 .svga，三态三个文件或一个文件多段，新增 `SvgaPetRenderer` 按 activity 切换并监听 `end`/`process` 做 strike。  
   - **VAP**：适合做功德爆发、升级等**特效**；若坚持用 VAP 做角色本体，也可三态三个 MP4，由 `VapPetRenderer` 切换并 loop。  
   - 新角色（lobster、cat、robot）**直接**用 Rive / Lottie / SVGA 做资产，不再写 SVG 骨骼代码。

3. **长期**  
   - 所有官方桌宠统一为 **Rive（或 Lottie）资产 + 同一套渲染管线**。  
   - `SvgBonePetRenderer` 仅用于「无 Rive/Lottie 时的回退」或历史包，不再扩展。

## 五、小结

| 项目 | 说明 |
|------|------|
| **最科学合理的方式** | 对桌宠而言**骨骼动画**最合适；骨骼方案不只有 Rive，还有 **Spine**（买断）、**DragonBones**（免费开源）等，见上文「二（补充）」对比。 |
| **为何不用手写 GSAP 做桌宠** | 层级/轴心/状态全写在代码里，易脱手、难复用、难和设计协作，且易漂移。 |
| **SVGA / VAP 可以吗** | **SVGA 可以**：与 Lottie 同思路，三态多文件或多段，免费、有 Web 播放器，适合做桌宠角色。**VAP 可以**：更适合复杂特效/粒子，做角色时相当于预渲染视频循环播放，无骨骼；可按需接。 |
| **项目现状** | 已支持 Rive / Lottie / SVG 三档；若要接 SVGA 或 VAP，需在 `pet-engine` 中增加类型并实现 `SvgaPetRenderer` / `VapPetRenderer`。 |
| **Adobe Animate 可以吗** | **可以**。Animate 自带骨骼/IK，做好的动画用 **SVGA 转换器** 导出 .svga 即可，官方支持 Animate CC；多态即多文件或多段，接 SVGA 播放与事件。 |
| **下一步** | 为 monk（及后续角色）制作 `.riv`、Lottie JSON 或 **.svga**（可用 Animate 做后转 SVGA），在 `pet-engine` 中配置资产路径即可切换，无需再改 `SvgBonePetRenderer`。 |

Rive 的接入细节见：[Rive动画方案与价格说明.md](./Rive动画方案与价格说明.md)、[Rive动画接入操作指南.md](./Rive动画接入操作指南.md)。

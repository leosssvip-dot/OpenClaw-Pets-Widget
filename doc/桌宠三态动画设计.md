# 四只桌宠 · 三态动画设计（统一 SVG + GSAP）

四个桌宠统一采用 **SVG 骨架 + GSAP 时间轴** 驱动三态动画（工作中、空闲、掉线），便于节奏可控、与 monk 方案一致。设计阶段优先；实现时再按本文拆 SVG、写 timeline。

## 已知问题（待修复）

- **Monk**：动画长时间运行后，手（arm）和棍子（mallet）会逐渐偏离原始位置，疑为 GSAP 的 transform 累积或 `svgOrigin` 漂移导致。
  **推荐修复方案**：用 `gsap.fromTo()` 替代 `gsap.to()`，固定每帧的起止值，根本上消除累积误差。`clearProps` 作为 fallback，在 `onRepeat` 回调中重置。

## 统一技术方案

| 项目 | 说明 |
|------|------|
| **渲染** | 每只桌宠一个 SVG，内部用 `class` 标记可动部件（如 `desktop-pet__monk-arm`、`desktop-pet__lobster-claw--left`）。 |
| **驱动** | 仅用 GSAP timeline，按状态切换不同 timeline（working / idle / offline），不用 CSS keyframes 驱动角色本体。 |
| **状态** | 工作中 = 快节奏、明确动作；空闲 = 慢节奏、呼吸/待机；掉线 = 静帧 + **必选灰化**（`filter: grayscale(0.4) brightness(0.92)`）。 |
| **GSAP 生命周期** | **必须**使用 `gsap.context()` 包住每只宠的 root，unmount 时 `context.revert()`，防止内存与全局 selector 残留。 |
| **无障碍** | 提供 `useReducedMotion()` hook；所有 timeline 构建函数接收 `reducedMotion: boolean` 参数；`prefers-reduced-motion` 时不跑 timeline 或只做极弱 opacity 呼吸。 |
| **class 命名** | 统一 `desktop-pet__{role}-{part}--{modifier}`，便于 GSAP 用 `root.querySelector()` 稳定选取。 |

### 状态来源与映射

| 来源 | 条件 | 动画状态 |
|------|------|----------|
| `connectionStatus` | `'offline'` 或 `'auth-expired'` | **掉线** |
| `petStatus` | `'disconnected'` | **掉线** |
| `petStatus` | `'working'` / `'thinking'` / `'collaborating'` | **工作中** |
| `petStatus` | `'idle'` / `'waiting'` / `'done'` 及其他 | **空闲** |

> 注意：`collaborating` 归入工作中，`done` 归入空闲（done 无自动衰减逻辑，由 gateway 事件驱动回 idle）。

### 状态过渡规则

- **working ↔ idle**：切换时先 `timeline.kill()`，再建新 timeline 并 `play()`。当前为硬切（无过渡），效果可接受。
- **→ 掉线**：`timeline.kill()` + 加灰化 CSS class。
- **掉线 →**：移除灰化 CSS class + 启动对应 timeline。
- 未来可选优化：用 0.2–0.3s 的 opacity 交叉过渡实现软切换（P2）。

## SVG 资产交付规范

SVG 由设计师提供或开发者从现有素材拆分，需满足以下约束：

- 每个可动部件必须是独立的 `<g>` 节点，并带有规范的 `class`
- 在 SVG 或代码注释中注明每个可动部件的 `transform-origin` / `svgOrigin`（如 `MONK_ARM_ORIGIN`），方便调参
- 不可动的装饰元素可合并为单一 `<g>`，减少 DOM 节点数
- 测试 checklist：每个 SVG 交付后，逐一对可动 class 做 `querySelector` 验证是否能选中

## SVG+GSAP 精细化实操指南（如何做更细的动画）

在已有「可动部件 + timeline」的基础上，要做得更细、更稳、更自然，按下面几步做即可。

### 1. SVG 拆得更细、原点固定

- **可动节点**：每个会动的地方都包成独立 `<g>` 或带 class 的元素，并加上 `desktop-pet__{role}-{part}--{modifier}`，方便 GSAP 用 `root.querySelector(...)` 稳定选取。
- **旋转/缩放原点**：在 GSAP 里对 SVG 做 `rotation` 时一定要设 `transformOrigin`（或 SVG 用 `svgOrigin`）。例如和尚的 `MONK_ARM_ORIGIN = '97 77'`、`MONK_MALLET_ORIGIN = '112 68'` 对应手臂/棍子的「关节」在 viewBox 里的坐标，这样旋转才不会跑偏。
- **做法**：在编辑器里打开 SVG，找到关节/支点像素坐标，在代码里用常量（如 `ARM_ORIGIN`）并在每次 `gsap.to(arm, { rotation: ..., svgOrigin: ARM_ORIGIN })` 里传入；新角色也先标好各部件原点再写 timeline。

### 2. 用 fromTo 固定起止，避免漂移

- **问题**：长时间循环时，用 `to()` 会基于「当前值」插值，浮点累积会导致手/棍子等慢慢偏离。
- **做法**：所有循环内的关键 pose 都用 `fromTo(target, { ...fromVals }, { ...toVals })` 明确起止，例如每周期开始 `fromTo(arm, { rotation: -5 }, { rotation: 2, duration: 0.12, ease: 'power3.in' })`，下一段再 `fromTo(arm, { rotation: 2 }, { rotation: -5, ... })`。这样每周期都从固定数值回到固定数值，不会漂。
- **配合**：每周期开头对易漂的节点做一次 `gsap.set(targets, { clearProps: 'transform' })` 或 `.set(targets, resetProps)` 再跑本周期动画，双保险。

### 3. 缓动选对，节奏分层

- **发力/撞击**：用 `power2.out`、`power3.in`、`power4.in`，让「动」的那一下有加速度感。
- **回弹/还原**：用 `sine.out`、`power1.out`，让收尾自然不弹跳（若要轻微 overshoot 用 `back.out(1.1)`）。
- **呼吸/待机**：用 `sine.inOut` 或 `power1.inOut`，整段柔和。
- **避免**：`ease: 'none'` 或 `linear` 只用在需要严格对齐时间的场合（如两段必须同时结束），其余尽量带一点缓动。

### 4. 次要动作错峰（跟随感）

- **做法**：同一时刻不要所有部件一起动。例如「敲击」：手臂先动 → 0.02–0.05s 后棍子/钳子动 → 再 0.02s 身体微倾；尾巴/耳朵可再晚 0.03s。在 timeline 里用不同 `to()` 的起始 position 错开，或对多个 target 用 `stagger: 0.03`。
- **效果**：主动作带一点「跟随」，观感更顺。

### 5. 接触瞬间的反馈

- **做法**：敲击/扫描碰到的那一帧，给被碰物体加一段极短 tween（如 0.06–0.1s）：`scale: 0.96` → 1，或 `opacity` / `filter: brightness()` 闪一下再恢复。
- **效果**：因果明确，点击感更强。

### 6. 空闲态不机械

- **多 phase**：一个 idle 周期拆成 2–3 段，例如「吸气 0.7s → 停 0.2s → 呼气 0.8s」，每段用不同 `duration` 和轻微不同幅度，避免完全对称的三角波。
- **偶尔小动作**：在主循环里加一个概率或每 N 周期触发一次的「耳朵动一下」「眨眼」：用 `timeline.add(() => { ... }, position)` 调一个子 timeline 或 `gsap.to(ear, { ... })`，或用 `repeatDelay` 让某段偶尔多停一会再继续。

### 7. 常量与可调参数集中

- **做法**：把各阶段的 duration、幅度（rotation、translate 的数值）、`svgOrigin` 都放到 `pet-timelines/constants.ts` 或各角色文件顶部的常量里，例如 `LIFT_DUR = 0.12`、`STRIKE_DUR = 0.08`、`ARM_RANGE = 5`。调手感时只改常数，不动 timeline 结构。
- **效果**：便于统一节奏、也便于做「工作中/空闲」两套参数（空闲 duration 拉长、幅度缩小）。

### 8. 无障碍：reducedMotion 降级

- **做法**：所有 `build*Timeline` 接收 `reducedMotion?: boolean`。为 `true` 时只跑极弱动画（如 4s 一次 opacity 0.98–1），或直接返回 `gsap.timeline({ paused: true })` 不播放。用 `prefers-reduced-motion: reduce` 的 media query 在组件里拿到后传入。
- **效果**：满足系统「减少动画」设置，不晕不闪。

按上述 8 点做，SVG+GSAP 方案就能在保持当前架构的前提下，做到更精细、更稳、更自然；细节程度可随工时逐步加（先 1–3，再加 4–6）。

## 多宠同屏性能

- 当前 `MultiPetShell` 支持多只桌宠同屏，每只跑独立 GSAP timeline。
- **性能预算**：同屏 ≤4 只为安全范围（4 条 GSAP timeline + 粒子系统）。超过时建议对屏幕外/最小化的宠暂停 timeline（`timeline.pause()`），可见时恢复。
- GSAP timeline 本身开销极低（仅操作 transform/opacity），瓶颈在粒子 DOM 节点数。

## 趣味性与成长体系设计（仿 Monk 功德体系）

目标：每只桌宠都有**可累积的成长点数**与**里程碑成就**，配合浮动粒子 + 达成时的庆祝 overlay，增强陪伴感与趣味性。Monk 现有实现作为模板，其余三只沿用同一套技术（store 按 role 或 role+petId 区分、粒子文案与里程碑按角色配置）。

### 计数规则

- **仅工作中状态计数**，空闲和掉线不计数，不产生粒子。
- 每个 timeline 周期完成时 +1（约 0.4–0.6s 一次），即工作中每小时约 6000–9000 次。
- 里程碑阈值参考（按持续工作时间估算）：

| 阈值 | 约需持续工作时间 | 体感 |
|------|-----------------|------|
| 10 | ~5 秒 | 即时反馈 |
| 100 | ~1 分钟 | 开始工作 |
| 1000 | ~8 分钟 | 一个小任务 |
| 10000 | ~1.4 小时 | 半天工作量 |
| 100000 | ~14 小时 | 持续积累 |

> 如希望里程碑更有"一天/一周"的节奏感，可调整为仅在 working timeline 的"关键帧"（如敲击落下瞬间）计数，而非每周期都计。

### 存储方案

- 沿用 `merit-store` 结构，key 为 `petId`（每只宠独立进度）。
- 通过 Electron 的 `writeSettings` / `readSettings` 持久化到本地。
- 暂不做多设备同步；未来可选同步到 gateway。

### Monk 现有体系（参考）

- **触发**：每次木鱼敲击（idle/working timeline 的 strike 时刻）触发一次计数。
- **粒子**：浮动「功德+1」，与敲击节奏同步，`MeritParticles` + `merit-store` 持久化。
- **里程碑**：10 初入佛门、50 小沙弥、100 功德圆满·百、500 禅心初成、1000 功德千秋、5000 大彻大悟、10000 万德庄严、50000 法力无边、100000 功德无量；达成时弹出 `MeritCelebration`（icon + label + 功德 N）。
- **展示**：桌宠旁常驻「功德 N」小计数（可带当前/下一档成就 icon），点击或悬浮可看下一档目标。

### 龙虾 (lobster) — 「修得」/ 码力成长

- **主题**：码字、修 bug、提交，程序员梗。
- **触发**：工作中 timeline 每次「敲键盘」动作完成时 +1（与 monk 敲击类似，一次循环一计数）。
- **粒子文案**：随机加权——70%「修得+1」、15%「bug-1」、15%「commit+1」；颜色偏绿/蓝。
- **成长单位名**：修得。
- **里程碑建议**：
  10 初出茅庐、50 小试牛刀、100 百行成诗、500 重构初成、1000 千行无悔、5000 开源在望、10000 万行一念、50000 代码如山、100000 功德无量（彩蛋，刻意与和尚对齐）。
- **趣味细节**：空闲时偶尔冒「待命中…」小气泡；达成里程碑时龙虾可做 1 秒「举钳庆祝」小动画（GSAP 单次）。

### 猫咪 (cat) — 「清单点」/ 规划成长

- **主题**：待办、清单、排期，规划向。
- **触发**：工作中 timeline 每次「点头/看便签」动作完成时 +1。
- **粒子文案**：随机加权——70%「清单+1」、15%「✓ 一项」、15%「待办-1」。
- **成长单位名**：清单点。
- **里程碑建议**：
  10 待办萌新、50 清单入门、100 百事俱备、500 排期有方、1000 千策在胸、5000 事无遗漏、10000 万单不侵、50000 规划大师、100000 功德无量（彩蛋）。
- **趣味细节**：便签上可显示当前进度条或下一个成就名（极简文案）；达成时便签飞出一张「成就」小卡片动画（或猫耳朵竖起来 1 秒）。

### 机器人 (robot) — 「巡检点」/ 运维成长

- **主题**：巡检、心跳、信号正常，运维向。
- **触发**：工作中 timeline 每次「扫描/监控」周期完成时 +1。
- **粒子文案**：随机加权——70%「巡检+1」、15%「心跳正常」、15%「信号✓」。
- **成长单位名**：巡检点。
- **里程碑建议**：
  10 值班新人、50 在岗熟练、100 百次巡检、500 稳定运行、1000 千日无虞、5000 故障克星、10000 万秒在线、50000 运维宗师、100000 功德无量（彩蛋）。
- **趣味细节**：状态条/顶灯颜色随成就档位微变（如 bronze 绿、silver 蓝、gold 金）；达成时屏幕闪一次「MILESTONE」或灯全亮 0.5s。

### 统一技术要点

- **存储**：沿用 `merit-store` 结构，key 为 `petId`；通过 `writeSettings` 持久化。
- **粒子组件**：复用 `MeritParticles`，通过 props 传入 `text`（如「修得+1」）、`role`；里程碑配置按 role 从 `growth-milestones.ts` 或现有 `merit-milestones` 扩展为 per-role。
- **庆祝**：复用 `MeritCelebration`，文案与 icon 按 role 和当前 milestone 配置；样式可保留 tier（bronze/silver/gold/diamond），仅文案与 icon 按角色替换。
- **趣味优先级**：先做「触发 + 粒子 + 里程碑 + 庆祝」与 Monk 对齐；「达成时桌宠小动画」「便签/屏幕上的进度展示」可作为二期增强。

## 动画精细化建议（P2）

- **缓动（easing）**
  - 工作中：发力动作用 `power2.out` / `power3.in`，回弹用 `sine.out`，避免 linear 显得生硬。
  - 空闲：统一用 `sine.inOut` 或 `power1.inOut`，呼吸感更自然。
  - 掉线：静帧为主，无需 easing。

- **次要动作与跟随**
  - 主动作带一点跟随：例如和尚敲击时身体先微前倾再回正，龙虾钳子落下时尾巴略滞后摆动，猫点头时耳朵略晚几帧。可用 timeline 内错开 0.02–0.05s 的 `stagger` 或错峰 `to()`。
  - 适当 overshoot：如机器人扫描到头时略过一点再回弹（`back.out(1.2)`），敲击类可加极小的回弹，避免完全线性收尾。

- **空闲态多样化**
  - 避免单一一拍循环显得机械：可做 2–3 段小 phase（如「呼吸上升 → 保持 → 呼吸下降」时长略不同），或在一个长周期内加一次「偶尔眨眼/耳朵动一下」的随机感（用 `repeatDelay` 或内嵌 sub-timeline）。
  - 和尚已有「呼吸 + 敲击」组合，其他宠也可考虑「主循环 + 偶尔小动作」，而不是完全对称的 loop。

- **节奏与幅度规范（建议值）**
  - 工作中：单次动作 0.08–0.15s（敲/点/扫），完整周期 0.4–0.6s；位移/旋转幅度小（约 2–6deg、1–5px），避免抢戏。
  - 空闲：单次呼吸 0.6–1.2s，完整周期 2.2–3s；幅度更小（约 1–3deg、0.5–2px）。
  - 掉线：静帧，不抢注意力。

- **可选细节增强**
  - **眨眼**：idle/working 共用的随机眨眼（如 3–5s 一次），可用独立 timeline 或 CSS 的短 keyframes，避免所有宠同时眨。
  - **掉线视觉**：龙虾/猫眯眼（`scaleY(0.6)` 或换 path）；机器人屏幕/灯常暗或 1.5s 一次弱闪，强化「断线」语义。
  - **接触反馈**：敲击/扫描的瞬间，被敲物体（木鱼、键盘、便签）可带 1 帧的 scale 或 glow 闪动，增强因果感。

- **无障碍实现**
  - 提供 `useReducedMotion()` hook（监听 `prefers-reduced-motion` media query）。
  - 所有 timeline 构建函数接收 `reducedMotion` 参数：为 `true` 时仅保留极弱 opacity 呼吸（4s 周期、opacity 0.97–1），禁用 rotation/translate。
  - 成长粒子在 `reducedMotion` 时也需降级：取消浮动动画，改为静态短暂显示后淡出。

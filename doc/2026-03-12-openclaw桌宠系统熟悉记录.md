# OpenClaw 桌宠系统熟悉记录

更新时间：2026-03-12

## 这是什么

这个仓库是一个以 macOS 为主的 OpenClaw 桌宠应用。它不是单纯的前端页面，而是一个 `Electron + React + TypeScript` 的桌面壳层应用：

- 桌宠本体以悬浮小窗形式运行
- 控制面板以独立窗口形式依附在桌宠旁边
- 通过 SSH 隧道连接另一台机器上的 OpenClaw Gateway
- 每个 agent 会被投影成一个 pet，在桌面上显示状态并接受交互

## 仓库结构

### 1. `apps/macos-shell`

真正的桌宠应用壳层，分为两部分：

- `electron/`
  - 负责窗口创建、IPC、SSH 隧道、密钥存取
- `src/`
  - 负责 React 渲染、桌宠交互、面板 UI、状态管理

### 2. `packages/bridge`

桥接层，负责：

- 连接 OpenClaw Gateway
- 维护 WebSocket 通信
- 请求 `agents.list`
- 发送 `agent.message.send` / `agent.task.create`
- 把网关事件转换成前端可消费的 `HabitatEvent`

### 3. `packages/domain`

共享领域模型，定义：

- `PetStatus`
- agent / pet 相关基础类型

它的职责比较轻，主要提供统一类型边界。

## 桌宠系统主入口

### Electron 入口

`apps/macos-shell/electron/main.ts`

这里负责：

- 创建 `petWindow` 和 `panelWindow`
- 根据 pet 窗口位置对齐 panel 窗口
- 暴露 IPC 能力给渲染层
- 接管 SSH 连接准备和销毁
- 接管安全存储

可以把它理解成“桌宠平台层”。

### React 入口

- `apps/macos-shell/src/main.tsx`
- `apps/macos-shell/src/App.tsx`

`App.tsx` 是整个桌宠系统的前端总控：

- 根据 query/runtime 判断当前是 `pet` 还是 `panel`
- 维护连接状态
- 管理 agent 列表和 pet 映射
- 连接 bridge 并订阅事件
- 决定渲染单宠还是多宠

## 关键运行流程

### 1. 应用启动

1. Electron 主进程创建两个窗口：
   - 桌宠窗口 `pet`
   - 控制面板窗口 `panel`
2. 两个窗口都加载同一个前端入口，但通过 `surface` 参数区分渲染模式
3. React `App` 根据 `surface` 决定显示桌宠还是面板

### 2. 建立远程连接

1. 用户在面板里填写 SSH/Gateway 配置
2. `App.tsx` 保存 profile 到 `settingsStore`
3. 渲染层通过 `runtime-deps.ts` 调用 `OpenClawClient`
4. `OpenClawClient.connect()` 会先调用桌面 API 的 `prepareGatewayConnection`
5. Electron 主进程中的 `SshTunnelRuntime` 建立本地到远程 Gateway 的 SSH 隧道
6. bridge 再通过本地 `ws://127.0.0.1:<port>` 连上 Gateway

这里的设计边界很清楚：

- 渲染层不知道 SSH 细节
- 桥接层不知道 Electron 窗口细节
- 主进程负责真正的系统能力

### 3. 获取 pet 列表

连接成功后：

1. bridge 调用 `agents.list`
2. `App.tsx` 把 agent 列表转成 habitat pets
3. `habitatStore` 存储当前 pets
4. `settingsStore` 保存 `petId -> agentId` 绑定关系

### 4. pet 状态更新

1. bridge 订阅网关事件
2. `parseOpenClawEvent()` 把原始事件转成统一的 `HabitatEvent`
3. `habitatStore.applyEvent()` 更新 pet 状态
4. `DesktopPet` 根据 `connectionStatus + petStatus` 解析出动画状态

当前已识别的核心状态包括：

- `idle`
- `thinking`
- `working`
- `waiting`
- `done`
- `blocked`

## 桌宠 UI 关键模块

### 1. `DesktopPet.tsx`

单个桌宠核心组件，负责：

- 点击打开气泡输入
- 双击切换面板
- 拖拽移动桌宠窗口
- 右键菜单
- 根据状态切换视觉表现

这是“桌宠交互核心”。

### 2. `MultiPetShell.tsx`

多宠模式容器：

- 当显示模式为 `group` 且 pet 数量大于 1 时启用
- 多个 `DesktopPet` 并排/错位展示

### 3. `WidgetPanel.tsx`

控制面板，负责：

- 连接状态显示
- 快速下发任务
- 切换 agent
- 切换角色皮肤
- 管理 gateway 配置

它不是系统核心逻辑层，更像“桌宠控制台”。

## 状态管理分工

### `habitatStore`

负责运行时状态：

- 当前 pets
- agent snapshots
- 当前选中的 pet
- pet 状态事件归并

### `settingsStore`

负责持久化配置：

- gateway profiles
- pet 与 agent 绑定
- 外观配置
- 显示模式
- pinned agent
- pet 窗口位置

这个分层是合理的：一个管“运行态”，一个管“配置态”。

### `widgetStore`

负责很轻量的界面状态：

- panel 是否打开

## 桌宠渲染方式

当前桌宠渲染主要还是 SVG/GSAP 路线：

- `DesktopPetIllustration.tsx`
- `RolePackIllustration.tsx`
- `pet-animation-state.ts`

项目里已经预留了 Rive 渲染抽象：

- `pet-engine.ts`
- `PetRenderer.tsx`
- `RivePetRenderer.tsx`

但从当前代码看，Rive 资产映射还是空的，现阶段实际仍以 SVG 方案为主，Rive 更像后续扩展位。

## 安全相关现状

目前安全设计里比较关键的点：

- Gateway token 和密码不会直接持久化到普通 settings
- 密钥通过 Electron 主进程的 `secure-store.ts` 存取
- 在可用时会使用 `safeStorage` 做系统级加密
- SSH 隧道在主进程建立，降低了渲染层直接接触系统能力的范围

需要注意的现实边界：

- 如果本机环境不支持 `safeStorage`，会退化为明文保存
- 当前日志里有 bridge 通信调试输出，后续如果进入生产阶段，需要重新检查日志是否会暴露敏感信息

## 我目前的结论

如果只从“整体结构和主流程”这个层级看，这个桌宠系统已经比较清晰：

- Electron 主进程负责平台能力
- React `App` 负责前端总控编排
- `bridge` 负责 OpenClaw 通信协议
- `habitatStore/settingsStore/widgetStore` 负责不同层次状态
- `DesktopPet` 和 `WidgetPanel` 分别承担桌宠本体与控制台 UI

换句话说，这不是“把一个角色画在桌面上”的小组件，而是一套：

- 有远程连接能力
- 有 agent 映射能力
- 有多宠显示能力
- 有状态驱动表现能力
- 有桌面壳层能力

的完整桌宠系统。
